## Сценарий 1: Регистрация и онбординг специалиста {#сценарий-1}

### 1.1 Регистрация как специалист → статус PENDING

**Статус: ✅ Реализовано, ⚠️ с нюансами**

- `POST /api/auth/register` с `role: 'SPECIALIST'` создаёт пользователя и связанный профиль `Specialist` в одной транзакции.
- Профиль специалиста создаётся со статусом по умолчанию (судя по схеме — `PENDING`).
- Автоматический логин после регистрации (в `AuthContext.register`) приводит к тому, что **статус пользователя сразу меняется с PENDING на ACTIVE** на строке 198 `auth.ts`: `...(user.status === 'PENDING' && { status: 'ACTIVE', emailVerified: true })`. Это корректно для пользователя, но статус профиля специалиста (`Specialist.status`) остаётся `PENDING` и не меняется.

### 1.2–1.5 Заполнение профиля

**Статус: ✅ Реализовано**

- `PUT /api/specialists/:id` принимает все нужные поля: `name`, `specialty`, `description`, `fullDescription`, `price`, `location`, `format`, `tags`, `education`, `certifications`, `experience`.
- Защита: только сам специалист или админ может обновить профиль.

### 1.6 Настройка расписания

**Статус: ✅ Реализовано, ⚠️ потенциальная проблема**

- `POST /api/specialists/:id/availability` принимает массив слотов и создаёт/обновляет их.
- Проблема: операции `findFirst` + `create/update` для каждого слота выполняются через `Promise.all` без транзакции и без блокировки. При конкурентных запросах возможно дублирование слотов.
- Обходной путь: `findFirst` ищет по `specialistId + date + time`, что является составным ключом — риск невысок, но теоретически присутствует.

### 1.7 Одобрение администратором

**Статус: ✅ Реализовано, ❌ уведомление не отправляется**

- `PUT /api/admin/specialists/:id/approve` обновляет статус до `APPROVED`.
- **BUG-12 (P1):** Функция `notifySpecialistApproved` в `services/notifications.ts` существует, но **нигде не вызывается** в `admin.ts`. После одобрения специалист не получает уведомление.
- Аналогично для `reject` — `notifySpecialistRejected` также не вызывается.

## Сценарий 3: Поиск и выбор специалиста {#сценарий-3}

### 3.1 Каталог специалистов

**Статус: ✅ Реализовано**

- `GET /api/specialists` возвращает только `APPROVED` специалистов.

### 3.2 Раздел "Для вас" (AI-рекомендации)

**Статус: ❌ Не реализовано**

- Нет эндпоинта для персонализированных рекомендаций на основе AI-консультации.

### 3.3 Фильтры

**Статус: ✅ Реализовано**

- Фильтры по цене, рейтингу, формату, тегам, локации — реализованы.
- ⚠️ Фильтр `minPrice`/`maxPrice` применяет `parseInt`, что обрежет дробные значения. Если цены хранятся в копейках — это нормально, если в рублях — может отсеять валидные значения с дробями.

### 3.4 Поиск по имени/специальности

**Статус: ✅ Реализовано**

- `search` параметр ищет по `name`, `specialty`, `description` через `OR`.
- Также есть отдельный `GET /api/search` с более широкими возможностями.

### 3.5 Сортировка

**Статус: ✅ Реализовано**

- Поддерживается сортировка по `price`, `rating`, `experience`.

### 3.6 Профиль специалиста

**Статус: ✅ Реализовано**

- `GET /api/specialists/:id` возвращает профиль со всеми полями, отзывами (только APPROVED) и слотами.
- Поддерживается просмотр своего собственного профиля даже без APPROVED статуса.

### 3.7 Доступные слоты (прошедшие скрыты)

**Статус: ✅ Реализовано**

- В `GET /api/specialists/:id` слоты фильтруются с `date: { gte: new Date() }`.
- **BUG-18 (P2):** `GET /api/specialists/:id/availability` (отдельный эндпоинт) **не фильтрует прошедшие даты** если не переданы явные `startDate`/`endDate`. Клиент может получить устаревшие слоты.

## Сценарий 4: Бронирование сессии {#сценарий-4}

### 4.1 Клиент выбирает слот → PENDING

**Статус: ✅ Реализовано, ❌ критический баг логики**

- `POST /api/bookings` создаёт бронирование в транзакции вместе с обновлением слота.
- **BUG-20 (P2) — Race Condition:** Проверка `timeSlot.isBooked: false` происходит до транзакции. При одновременных запросах от двух клиентов оба пройдут проверку, после чего транзакция выиграет один из них, а второй получит ошибку constraint только если есть уникальный индекс. Если индекса нет — оба слота будут помечены как `isBooked: true` и оба бронирования будут созданы для одного слота. Требуется проверка внутри транзакции или использование `SELECT FOR UPDATE`.

### 4.2 Клиент добавляет сообщение специалисту

**Статус: ✅ Реализовано**

- Поле `clientMessage` поддерживается в `POST /api/bookings`.

### 4.3 Уведомление специалиста о новом бронировании

**Статус: ⚠️ Частично реализовано, ❌ критическая логическая ошибка**

- **BUG-06 (P1):** В `bookings.ts` при вызове `notifyBookingCreated`, первым аргументом `clientId` передаётся `userId` (ID пользователя), а вторым `specialistId` — тоже `userId` специалиста. Код:

```typescript
await notifyBookingCreated({
  clientId: userId,           // userId клиента — правильно
  specialistId: specialist.userId,  // userId специалиста — правильно
  ...
})
```

Функция `notifyBookingCreated` в `services/notifications.ts` создаёт уведомление **и для клиента, и для специалиста**. Это поведение корректно. Однако в сценарии, когда **специалист сам создаёт бронирование** (строки 37–69 `bookings.ts`), `userId` является ID специалиста, и уведомление будет отправлено специалисту как "клиенту" вместо реального клиента. Это ошибочная логика при создании бронирования специалистом.

### 4.5 Специалист подтверждает → CONFIRMED

**Статус: ✅ Реализовано, ⚠️ неверная логика оплаты**

- `PUT /api/bookings/:id/status` с `status: CONFIRMED` обновляет статус.
- **BUG-07 (P1):** При подтверждении статуса, автоматически выставляется `isPaid: true` (строка 409): `...(status === 'CONFIRMED' && { isPaid: true })`. Это означает, что **подтверждение бронирования без оплаты помечается как оплаченное**. Логика должна быть обратной — `isPaid` должно выставляться только при подтверждении платежа через вебхук.

### 4.6 Клиент получает подтверждение

**Статус: ✅ Реализовано**

- `notifyBookingConfirmed` вызывается в `PUT /:id/status` при `status === 'CONFIRMED'`.

### 4.7 Страница "Мои записи"

**Статус: ❌ Критическое несоответствие типов данных**

- **BUG-08 (P1):** `Bookings.tsx` использует интерфейс `Booking` с полями `booking.service.title`, `booking.service.duration`, `booking.service.price`, `booking.startTime`, `booking.endTime`. **Ни одного из этих полей нет в ответе** `GET /api/bookings`. API возвращает `booking.specialist.name`, `booking.specialist.price`, `booking.date`, `booking.time`. Страница будет падать или показывать `undefined` для всех записей.
- Дополнительно: `otherPerson.user.firstName` — API возвращает `specialist.name` (строку), а не `specialist.user.firstName`. Аналогичная проблема для клиентской стороны.

## Сценарий 5: Оплата {#сценарий-5}

### 5.1–5.2 Переход к YooKassa и оплата

**Статус: ❌ Не реализовано (заглушка)**

- **BUG-03 (P0):** В `payments.ts` строки 110–116 содержат заглушку с `paymentUrl = \`https://yookassa.ru/checkout/${transaction.id}\``. Реальный вызов API YooKassa закомментирован. Пользователь будет перенаправлен на несуществующий URL.
- Переменные `YOOKASSA_SHOP_ID` и `YOOKASSA_SECRET_KEY` объявлены в `.env.example`, но не используются в коде.

### 5.3 Возврат на платформу → обновление статуса

**Статус: ⚠️ Webhook реализован, ❌ критическая уязвимость безопасности**

- **BUG-02 (P0):** `POST /api/payments/webhook` не проверяет подпись запроса (заголовок `X-Signature` от YooKassa). Комментарий в коде это подтверждает: "В реальном приложении нужна проверка подписи". Без проверки любой может вызвать вебхук и установить оплату для любой транзакции.
- Дополнительно: вебхук обрабатывает только `event === 'payment.succeeded'`, не обрабатывает `payment.canceled`, `refund.succeeded`.

### 5.4 История платежей

**Статус: ❌ Баг в интерфейсе**

- **BUG-09 (P1):** `Payments.tsx` делает запрос к `/api/payments` и читает `data.payments`. API возвращает `{ transactions: [...] }`. Страница будет всегда показывать пустой список.
- Дополнительно: `Payments.tsx` ожидает `payment.booking.service.title` и `payment.booking.specialist.user.firstName`, но API возвращает `transaction.booking.{date, time, status}` и `transaction.specialist.name`. Страница будет падать при отображении.

### 5.5 Баланс специалиста (минус 15% комиссия)

**Статус: ✅ Логика верна в вебхуке, ⚠️ несоответствие единиц**

- В вебхуке: `specialistAmount = transaction.amount - transaction.platformFee`. Это правильно.
- **BUG-16 (P2):** В `notifyPaymentReceived`: сообщение формирует `data.amount / 100` для отображения суммы в рублях, предполагая, что `amount` в копейках. Но в `payments.ts` при создании транзакции `amount = booking.price` без умножения на 100. Если `price` специалиста хранится в рублях, деление на 100 покажет неверную сумму (например, `3000 / 100 = 30` вместо `3000 ₽`). Вдобавок, сам `notifyPaymentReceived` нигде не вызывается в вебхуке — специалист не получает уведомление об оплате.

## Сценарий 6: Проведение сессии {#сценарий-6}

### 6.1–6.2 Чат в реальном времени

**Статус: ⚠️ Частично реализовано**

- Базовая отправка/получение сообщений через REST API + WebSocket-эмит реализована.
- **BUG-10 (P1):** `Messages.tsx` — основная страница чата — **не подписывается на WebSocket-события** `chat:message:new`. Сообщения обновляются только через `loadMessages()` (перезапрос к API) после каждой отправки. Входящие сообщения от другого участника не отображаются в реальном времени без перезагрузки страницы.
- `ChatWindow.tsx` использует `useChat` hook, который, вероятно, подписывается на сокет, но `Messages.tsx` не использует `ChatWindow`.

### 6.3 Индикатор "печатает..."

**Статус: ⚠️ Реализовано в ChatWindow, ❌ отсутствует в Messages.tsx**

- WebSocket-хендлер `chat:typing` в `chat.ts` (backend) и `ChatWindow.tsx` поддерживают typing-индикатор.
- **BUG-14 (P2):** `Messages.tsx` не вызывает `chat:join` для WebSocket-комнаты и не слушает `chat:typing`. Typing-индикатор не работает на основной странице сообщений.

### 6.4 Статус прочтения

**Статус: ⚠️ Частично реализовано**

- `ChatWindow.tsx` автоматически вызывает `markAsRead` для последнего сообщения.
- **BUG-22 (P3):** В `ChatWindow.tsx` `recipientId` берётся из `room.specialist.id` или `room.client.id`, которые являются ID таблиц `Specialist`/`Client`, а не `User.id`. API `sendMessage` (`POST /api/chat/messages`) ожидает `recipientId` как `User.id` (для поиска пользователя через `prisma.user.findUnique`). При отправке сообщения будет ошибка "Recipient not found".

### 6.5 Специалист завершает сессию как COMPLETED

**Статус: ✅ Реализовано**

- `PUT /api/sessions/:id` с полем `endedAt` автоматически меняет `booking.status = 'COMPLETED'` в транзакции и инкрементирует `specialist.totalSessions`.

### 6.6 Заметка специалиста о клиенте

**Статус: ✅ Реализовано**

- Поле `specialistNotes` в `PUT /api/sessions/:id`.
- Защита от утечки: в `GET /api/sessions` и `GET /api/sessions/:id` для клиентов поле `specialistNotes` удаляется из ответа.

## Сценарий 7: Отзыв и рейтинг {#сценарий-7}

### 7.1–7.4 Создание отзыва

**Статус: ✅ Реализовано**

- `POST /api/reviews` проверяет: пользователь — клиент, бронирование завершено (`COMPLETED`), бронирование принадлежит клиенту.
- Принимает `rating`, `text`, `professionalismRating`, `empathyRating`, `resultRating`.

### 7.5 Видимость отзыва (после модерации)

**Статус: ❌ Баг — автоматическое одобрение**

- **BUG-15 (P2):** В `POST /api/reviews` отзыв создаётся с `status: 'APPROVED'` (строка 84 `reviews.ts`). Комментарий: "Автоматическое одобрение (можно изменить на PENDING для модерации)". Это противоречит требованию сценария 7.5, где отзыв должен проходить модерацию.

### 7.6 Пересчёт рейтинга специалиста

**Статус: ✅ Реализовано**

- При создании, обновлении и удалении отзыва рейтинг пересчитывается в транзакции по всем APPROVED отзывам.

### 7.7 Запрет второго отзыва

**Статус: ✅ Реализовано корректно**

- Проверка: `prisma.review.findFirst({ where: { clientId, specialistId } })`. Уникальность по паре (клиент, специалист), а не по конкретному бронированию. Это значит, что клиент не может оставить отзыв другому специалисту если уже оставил один этому же — что является правильным поведением.

## Сценарий 8: Уведомления — детальный разбор известного бага {#сценарий-8}

### Известный баг: "Счётчик показывает непрочитанные, но список пуст"

**Статус: ❌ Подтверждён. Найдено несколько причин.**

#### Корневая причина 1 (P0) — двойной источник правды для счётчика и списка

Система использует два независимых пути к состоянию уведомлений:

**Путь A (REST API) — используется для начальной загрузки списка:**
```
GET /notifications → фильтр: { userId, type: 'IN_APP' }
GET /notifications/unread-count → фильтр: { userId, type: 'IN_APP', isRead: false }
```

**Путь B (WebSocket) — используется для счётчика в хуке `useNotifications`:**
```
socket.emit('notifications:get-unread-count')
→ backend: prisma.notification.count({ where: { userId, readAt: null } })
```

Заметьте: REST API проверяет `isRead: false`, WebSocket-хендлер проверяет `readAt: null`. Это **разные поля**.

#### Корневая причина 2 (P1) — `mark-all-read` через WebSocket не устанавливает `isRead`

**BUG-04:** В `handlers/notifications.ts`, функция `notifications:mark-all-read`:
```typescript
await prisma.notification.updateMany({
  where: { userId, readAt: null },
  data: { readAt: new Date() }   // <-- устанавливает только readAt
  // isRead НЕ устанавливается в true!
})
```

После `mark-all-read` через WebSocket поле `isRead` остаётся `false`. Следующий вызов `GET /notifications` вернёт те же записи с `isRead: false`. Список заполняется, счётчик через REST API снова показывает непрочитанные.

#### Корневая причина 3 (P1) — рассинхронизация счётчика

**BUG-05:** При вызове `notifications:mark-read` через WebSocket:
```typescript
data: { readAt: new Date() }   // isRead по-прежнему не меняется
```

При следующем вызове `notifications:get-unread-count` (WebSocket), фильтр `readAt: null` вернёт правильно обновлённый счёт. Но если фронтенд использует `GET /notifications/unread-count` (REST), тот вернёт старый счёт т.к. смотрит на `isRead: false`.

#### Корневая причина 4 (P2) — счётчик не обновляется при открытии дропдауна

**BUG-13:** В `NotificationsDropdown.tsx`:
- `useNotifications` запрашивает счётчик через WebSocket один раз при подписке.
- При открытии дропдауна и чтении уведомлений через REST API (`getNotifications`) — `unreadCount` в хуке не синхронизируется.
- Если уведомление пришло через WebSocket — счётчик растёт. Если список загружен через REST и отображает 10 прочитанных уведомлений, счётчик всё равно может показывать 10, т.к. REST-ответ не обновляет `unreadCount` в хуке.

#### Итоговая схема рассинхронизации

```
Создание уведомления:
  → DB: notification.isRead = false, notification.readAt = null
  → WS-счётчик: +1 (читает readAt = null) ✅

Пользователь открывает дропдаун:
  → REST загрузка: получает уведомления с isRead=false ✅
  → Отображает синие точки (непрочитанные) ✅

Пользователь нажимает "Прочитать все" (handleMarkAllAsRead):
  → WS-эмит: notifications:mark-all-read
  → DB update: readAt = new Date() (isRead ОСТАЁТСЯ false)
  → WS-ответ: notifications:all-read → local state: isRead=true ✅
  → WS-счётчик: 0 (читает readAt=null — нет таких) ✅

Пользователь закрывает и снова открывает дропдаун:
  → REST загрузка: select where isRead=false → ВОЗВРАЩАЕТ ВСЕ ЗАПИСИ
  → Список снова полный ❌
  → unreadCount через WS: 0 (readAt не null) ✅
  → Счётчик 0, список полный → видим уведомления, но без счётчика ❌
```

### 8.1–8.5 Остальные шаги сценария

| Шаг | Статус | Комментарий |
|-----|--------|-------------|
| 8.1 Счётчик непрочитанных | ❌ Сломан | BUG-01, BUG-04, BUG-05 |
| 8.2 Список не пустой | ❌ Сломан | После "прочитать все" список снова загружается с isRead=false |

## Сценарий 9: Отмена бронирования {#сценарий-9}

### 9.1 Отмена → CANCELLED

**Статус: ⚠️ Два способа отмены, один сломан**

Существует два пути отмены:
1. `DELETE /api/bookings/:id` — корректная отмена с `cancelledBy`, `cancellationReason`, освобождением слота и уведомлением второй стороны.
2. `PUT /api/bookings/:id/status` с `{ status: 'CANCELLED' }` — также работает, но **BUG-11 (P1):** не отправляет уведомление второй стороне и не устанавливает `cancelledBy`, `cancelledAt`.

**BUG-11:** `Bookings.tsx` использует именно второй путь (`PUT /bookings/:id/status`), что означает:
- Вторая сторона не получает уведомление об отмене.
- Поле `cancellationReason` не передаётся (нет UI для ввода причины в `Bookings.tsx`).

### 9.2 Причина отмены

**Статус: ❌ В UI отсутствует**

- `Bookings.tsx` вызывает `handleCancelBooking` с простым `confirm()` диалогом без поля ввода причины.

### 9.3 Уведомление другой стороны

**Статус: ❌ Не работает** (BUG-11)

### 9.4 Фильтр "Отменённые"

**Статус: ⚠️ Реализован на фронтенде, но с проблемой**

- Фильтрация по `booking.status === 'CANCELLED'` делается на стороне клиента из уже загруженных данных.
- Запрос к API отправляет `clientId`/`specialistId` как параметры, которые API **не использует** (API использует токен авторизации для определения пользователя, игнорируя `clientId`/`specialistId` в query).

**Статус: ❌ Не реализован**

## Кросс-сценарные проблемы {#кросс-сценарные-проблемы}

### Безопасность

1. **WebSocket-токен не обновляется**: `SocketContext` инициализируется один раз с токеном из localStorage. Если токен истекает — соединение обрывается без переподключения с новым токеном. Однако нет механизма обновления токена (refresh token) в системе.

2. **BUG-21 (P3)**: `SocketContext` запускается один раз при монтировании (`useEffect` с пустым массивом зависимостей). Если пользователь не был авторизован при загрузке страницы — WebSocket не подключится даже после логина.

3. **Отсутствие rate limiting**: Нет видимых ограничений на количество запросов к `/api/auth/login` (брутфорс паролей), `/api/payments/webhook` (спам вебхуками).

### Целостность данных

4. **BUG-25 (P3)**: В `PUT /api/admin/bookings/:id/cancel` освобождение тайм-слота происходит по условию `{ specialistId, date, time }`. Если у специалиста есть несколько бронирований на одно время (теоретически невозможно, но как защита) — это может освободить неправильный слот. Надёжнее использовать `{ bookingId: id }` (как в `DELETE /api/bookings/:id`).

### UX / Фронтенд

5. **Навигация из уведомлений**: `NotificationsDropdown.tsx` использует `window.location.href = notification.actionUrl` вместо React Router `navigate()`. Это вызывает полную перезагрузку страницы.

6. **`Bookings.tsx` запрашивает clientId/specialistId как query-параметры**: API их игнорирует, определяя пользователя по JWT токену. Это не баг, но лишний код, вводящий в заблуждение.
